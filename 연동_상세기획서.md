# 광명 스마트데이터 플랫폼 — 보드 ↔ 포털 연동 상세 기획서

작성일: 2026-07-07

---

## 0. 핵심 결론 먼저 — "다른 언어인데 연동·확인이 가능한가?"

**가능합니다.** 두 사이트 모두 최종적으로는 브라우저에서 HTML/JS로 렌더링되므로, 스택이 달라도 표준 웹 방식(링크·`window.open`·URL 파라미터·세션 저장소)으로 자유롭게 연동됩니다. 실제 스택을 확인한 결과는 다음과 같습니다.

| 구분 | smartdataboard (관리자/운영) | smartdataportal (시민 서비스) |
|---|---|---|
| 성격 | 운영자용 데이터 대시보드 | 시민 공개 포털 |
| 스택 | **정적 HTML + Vanilla JS + CSS** (멀티 페이지) | **React 19 + Vite + TypeScript** SPA |
| 라우팅 | 페이지별 `.html` 파일 | 상태값 `currentPage` (URL 라우터 없음) |
| 로그인 | `login.html` (자체 로그인, 노출됨) | `Login.tsx` (**현재 주석 처리로 숨김**, 폴더에 존재) |
| 인증 저장 | `sessionStorage['gmsb-auth']` | (아직 없음 — 신규 구축 필요) |
| 관리자 화면 | 대시보드 전체가 관리자 화면 | `Admin.tsx` (콘텐츠·계정 CMS) |
| 배포 | 정적 호스팅 (파일만 업로드) | Vite 빌드 → 정적 배포 (현재 Vercel) |

> 즉, 요구사항에서 말한 "숨겨진 로그인 페이지"는 **포털의 `Login.tsx`** (App.tsx에서 주석 처리됨)이고, "숨겨야 할 상단 관리자 메뉴"와 "삭제할 소셜 로그인"도 모두 **포털 쪽**에 실재함을 코드로 확인했습니다. 기획 방향과 실제 코드가 정확히 일치합니다.

## 1. 요구사항 정리

| # | 요구사항 | 대상 |
|---|---|---|
| R1 | smartdataboard 내 특정 지점 클릭 → **포털 관리자 로그인 화면**으로 진입 | board → portal |
| R2 | 포털 **상단 GNB의 "관리자" 메뉴 숨김** (시민에게 미노출) | portal |
| R3 | 보드에서 포털에 로그인 성공 시 → **포털 관리자 페이지가 새 창으로** 오픈 | portal |
| R4 | 포털 로그인 화면에서 **네이버·카카오 등 소셜 로그인 삭제**, 일반(ID/PW) 로그인만 유지 | portal |
| R5 | 서로 다른 스택 간 연동 가능 여부 확인 | (0장에서 완료) |

## 2. 전체 연동 플로우

```
[운영자, smartdataboard 로그인 상태]
        │
        │  ① 보드 상단 계정 드롭다운의 "포털 관리자 콘솔" 클릭
        ▼
[smartdataportal 로그인 화면(Login.tsx)]   ← 시민 GNB에서는 관리자 메뉴 숨김(R2)
        │  ID/PW만 입력 (소셜 로그인 제거, R4)
        │  ② 로그인 성공
        ▼
[포털 관리자 페이지(Admin.tsx)를 새 창(target=_blank)으로 오픈]  (R3)
        │
        ▼
[관리자 CMS: 콘텐츠 관리 / 관리자 계정 / 대시보드]
```

포털의 유일한 관리자 진입 경로를 "보드 → 로그인"으로 단일화합니다. 시민은 GNB에서 관리자 메뉴 자체를 볼 수 없으므로(R2), 관리자 영역이 자연스럽게 은닉됩니다.

## 3. 배포/오리진 모델 — ✅ 확정: 다른 오리진(다른 URL) 허용

**서로 다른 도메인(다른 URL)이어도 이번 요구사항은 100% 정상 동작합니다.** 확정된 플로우가 **세션 공유를 필요로 하지 않기** 때문입니다. (포털 홈=로그인 불필요, 보드=자체 로그인, 포털 관리자=자체 프런트 검증 → 각자 독립 로그인)

| 동작 | 다른 오리진일 때 |
|---|---|
| 보드 → 포털 링크 열기 (`window.open`/`target=_blank`) | ✅ 정상 |
| 포털이 `?page=login` 등 URL 파라미터 읽기 | ✅ 정상 (오리진 무관) |
| `localStorage`/쿠키 **공유** | ❌ 불가 — **단, 이번 플로우에 불필요** |

- 예) 포털 = `...vercel.app`, 보드 = 별도 호스팅 → 그대로 진행 가능.
- 코드에서는 포털 URL을 상수 `PORTAL_BASE_URL` 하나로 관리 → 배포 확정 시 값만 교체.
- **동일 오리진은 향후 진짜 SSO(보드 1회 로그인 → 포털 관리자 자동 인증)가 필요해질 때만** 재검토. 현 요구사항엔 이점 없음.

## 4. 세부 작업 항목 (파일 단위)

### R2 — 포털 상단 "관리자" 메뉴 숨김
- 파일: `smartdataportal/src/components/Header.tsx`
- 작업: 데스크톱 GNB의 `<a href="#admin">관리자</a>` 블록과, 모바일 드로어의 동일 `관리자` 항목 2곳을 **주석 처리(또는 조건부 렌더 플래그로 숨김)**.
- 참고: `onNavigate('admin')` 경로 자체는 유지(신규 창 진입에 사용). 메뉴 UI만 숨김.

### R4 — 포털 로그인 소셜 로그인 삭제
- 파일: `smartdataportal/src/components/Login.tsx`
- 작업: `{/* 또는 구분선 */}` 블록 + `{/* 소셜 로그인 */}` 의 카카오·네이버 버튼 전체 삭제. `MessageCircle` 등 미사용 import 정리.
- 결과: ID / 비밀번호 / 아이디 저장 / 로그인 버튼 / 하단 링크(아이디·비밀번호 찾기·회원가입)만 유지.

### R1 — 보드 → 포털 로그인 진입점 추가
- 파일: `smartdataboard/assets/js/shell.js` (상단바 계정 드롭다운 렌더 영역)
- 작업: 계정 드롭다운(`tb__user-drop`)에 항목 1개 추가 (**✅ 확정: 계정 드롭다운 방식**) —
  `<a class="tb__drop-item" href="{PORTAL_BASE_URL}/?page=login" target="_blank" rel="noopener">포털 관리자 콘솔</a>`

### R1/R3 — 포털: URL 진입 + 로그인 성공 → 관리자 새 창
포털은 현재 URL 라우팅이 없고 `currentPage` 상태로만 화면을 전환합니다. 아래 3가지를 신규 구축합니다.

1. **URL 파라미터 부팅** — `smartdataportal/src/App.tsx`
   - 초기 `currentPage`를 `?page=` 값으로 설정. 허용: `login`, `admin`.
   - 주석 처리된 `Login` import와 `currentPage === 'login'` 렌더 분기 복구.
2. **로그인 성공 핸들러** — `smartdataportal/src/components/Login.tsx`
   - 로그인 버튼/Enter 시: (프로토타입) ID/PW 검증 후
     - 관리자 세션 플래그 저장: `sessionStorage['sdp-admin-auth'] = JSON.stringify({...})`
     - **관리자 새 창 오픈**: `window.open(`${PORTAL_BASE_URL}/?page=admin`, '_blank', 'noopener')` (R3)
3. **관리자 접근 가드** — `App.tsx` 또는 `Admin.tsx`
   - `currentPage === 'admin'` 진입 시 `sdp-admin-auth` 없으면 자동으로 `login`으로 전환(직접 URL 접근 차단).

> 프로토타입 기준으로 로그인 검증은 프런트 데모 계정으로 처리(보드의 `admin/1234` 패턴과 동일 수준). 실제 서비스 전환 시 이 지점에 백엔드 인증 API를 연결하면 됩니다.

## 5. 세션·인증 연동 방식

- **보드 인증**: `sessionStorage['gmsb-auth']` (기존 유지). `shell.js`의 `checkAuth()`가 없으면 `login.html`로 리다이렉트.
- **포털 관리자 인증(신규)**: `sessionStorage['sdp-admin-auth']`. Login 성공 시 세팅, Admin 가드에서 확인.
- **두 세션은 독립**입니다(별도 시스템). 요구사항이 "포털에서 다시 로그인"을 명시하므로 별도 로그인 유지가 기본.
- (선택·향후) **SSO/원클릭**: 진짜 자동 인증이 필요해지면, 동일 오리진 전환 후 세션 공유 또는 서명 토큰 URL 전달로 확장. 현 요구사항 범위 밖.

## 6. 확정된 결정사항 (2026-07-07)

| # | 항목 | 확정 |
|---|---|---|
| 1 | 배포 오리진 | **다른 오리진(다른 URL) 허용** — 세션 공유 불필요, 링크·URL 파라미터로 연동 |
| 2 | 보드 진입점 | **상단바 계정 드롭다운** 메뉴 항목 |
| 3 | 로그인 검증 | **프로토타입 프런트 검증** (백엔드 API는 추후 전환) |
| 4 | "새 창 관리자" 정의 | **포털 `Admin.tsx`** (콘텐츠·계정 CMS) |

### URL 직접 진입 시 기대 동작 (확정)
- **포털 URL 직접 입력** → **홈 화면**(로그인 불필요, 공개). *(현재 이미 이렇게 동작함 — 변경 없음)*
- **보드 URL 직접 입력** → **로그인 화면**. *(현재 `shell.js`의 `checkAuth()`가 `gmsb-auth` 없으면 `login.html`로 리다이렉트 — 이미 충족)*
- **보드에서 "포털 관리자 콘솔" 클릭 → 포털 로그인 → 성공 시 포털 관리자 화면(새 창).**

## 7. 로컬 검증 방법 (스택이 달라도 함께 테스트 가능)

```bash
# 포털 (React/Vite) — http://localhost:3000
cd smartdataportal
npm install
npm run dev

# 보드 (정적 HTML) — 정적 서버로 서빙 (sessionStorage/리다이렉트 때문에 file:// 대신 http 권장)
cd smartdataboard
npx serve .        # 또는 python -m http.server
```

통합 시나리오 테스트: 보드 로그인(admin/1234) → 계정 드롭다운 "포털 관리자 콘솔" 클릭 → 포털 로그인 화면 → 로그인 → 관리자 페이지 새 창 오픈까지 end-to-end 확인.
